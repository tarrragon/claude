# 工作日誌撰寫方法論

**版本**: v1.0.0
**建立日期**: 2026-01-10
**核心原則**: 工具相容性優先，純文字狀態標記

---

## 方法論目的

定義工作日誌的撰寫規範，確保：
1. **格式一致性** - 所有工作日誌遵循統一結構
2. **工具相容性** - 避免 CLI 處理問題導致的 crash
3. **可維護性** - 便於搜尋、解析和自動化處理

### 背景

Claude Code CLI 在處理 markdown 表格中的特定 emoji 時會發生 Rust panic：
```
byte index 2 is not a char boundary; it is inside '⏳' (bytes 0..3) of `⏳ |`
```

此問題源於 Rust 字串切割時遇到多位元組 Unicode 字元的 byte boundary 問題。

---

## 第一原則：工具相容性

### 禁止項目

在 **markdown 表格單元格**中：
- 禁止使用 emoji 字元（如 `⏳`, `🔄`, `❌`, `🚫`）
- 禁止使用其他多位元組特殊符號作為狀態標記

**錯誤範例**：
```markdown
| Ticket ID | 狀態 |
|-----------|------|
| W1-001    | ⏳   |  <!-- 禁止：會導致 CLI crash -->
| W1-002    | 🔄   |  <!-- 禁止 -->
```

### 允許項目

- 表格外的段落中使用 emoji（標題、說明文字、列表項目）
- 純文字狀態標記

**正確範例**：
```markdown
| Ticket ID | 狀態 |
|-----------|------|
| W1-001    | 待處理 |  <!-- 正確：純文字 -->
| W1-002    | 進行中 |  <!-- 正確 -->

#### W1-001 分析結果：已完成

上述 Ticket 已於 2026-01-10 完成。  <!-- 表格外可用 emoji -->
```

---

## 第二原則：狀態符號標準化

### 表格內狀態（純文字）

| 狀態語意 | 標準寫法 | 禁用符號 |
|----------|---------|----------|
| 等待處理 | `待處理` | `⏳` |
| 正在執行 | `進行中` | `🔄` |
| 執行完畢 | `已完成` | - |
| 主動取消 | `取消` | `❌` |
| 跳過不做 | `跳過` | `⏭️` |
| 被阻塞中 | `阻塞` | `🚫` |
| 失敗錯誤 | `失敗` | `💥` |

### 表格外狀態（可用 emoji）

在段落、標題、列表中可使用：
- `已完成` 項目標記
- `失敗` 或取消項目標記
- `警告` 說明

**範例**：
```markdown
#### W2-001 分析結果：已完成

**核心發現**：
- 測試卡住問題不存在，所有 14 個測試 100% 通過
- 技術債務等級：無債務
```

---

## 第三原則：標準結構

### 工作日誌必要章節

```markdown
# vX.Y.Z [版本名稱] 工作日誌

**版本**: vX.Y.Z
**開始日期**: YYYY-MM-DD
**狀態**: 進行中 / 已完成
**分支**: `feature/xxx` 或 `fix/xxx`

---

## 版本目標

[簡要描述本版本要達成的目標]

---

## Phase 1: [階段名稱]

### [子任務或 Ticket 清單]

| Ticket ID | Action | Target | Agent | 狀態 |
|-----------|--------|--------|-------|------|
| X.Y.Z-W1-001 | [動詞] | [目標] | [代理人] | 待處理 |

---

## 技術筆記

[開發過程中的技術發現和決策記錄]

---

## 參考文件

[相關文件連結]
```

### Ticket 表格標準格式

| 欄位 | 說明 | 範例 |
|------|------|------|
| Ticket ID | 版本號-階段-序號 | `0.25.0-W1-001` |
| Action | 動詞（Analyze/Design/Fix/Implement） | `Analyze` |
| Target | 修改目標 | `Widget 測試基礎設施` |
| Agent | 執行代理人 | `sage-test-architect` |
| Dependencies | 依賴的其他 Ticket | `W2-001` |
| 狀態 | 純文字狀態 | `待處理` |

---

## 驗證檢查清單

撰寫工作日誌時，確認以下項目：

- [ ] 所有表格單元格使用純文字狀態
- [ ] 無 `| ⏳ |`、`| 🔄 |`、`| ❌ |` 等模式
- [ ] 版本資訊完整（版本號、日期、狀態、分支）
- [ ] Ticket ID 格式正確（版本號-階段-序號）
- [ ] 結構符合標準模板

---

## Hook 自動驗證

本方法論由 `worklog-format-check.py` Hook 自動驗證：

- **觸發時機**：PostToolUse (Edit/Write)
- **檢測範圍**：`docs/work-logs/**/*.md`
- **檢測內容**：表格中的問題 emoji 模式
- **行為**：警告（非阻擋），輸出問題位置

## 結論

工作日誌是開發過程的重要記錄，必須確保：
1. **純文字狀態** - 表格內禁用 emoji，使用標準化純文字
2. **結構一致** - 遵循標準模板，便於搜尋和解析
3. **自動驗證** - 依賴 Hook 系統進行格式檢查

遵循本方法論可避免工具相容性問題，並提升工作日誌的可維護性。
