# 程式碼自然語言化撰寫方法論

## 核心概念

程式碼是給人閱讀的業務邏輯文件，不是給機器的指令縮寫。

## 五大原則

### 原則一：自然語言命名

| 元素 | 規則 | 範例 |
|-----|------|------|
| 函式 | 動詞 + 完整目標 | `calculateBookReadingProgress()` |
| 變數 | 內容物描述，無縮寫 | `authenticatedUser`（非 `usr`） |
| 類別 | 業務職責描述 | `BookMetadataEnrichmentService`（非 `Manager`） |

### 原則二：五行函式

- 每個函式 **5-10 行**
- 超過 10 行 → 必須拆分
- 名稱使用「和/或」→ 必須拆分

### 原則三：變數職責專一

- 每個變數只承載 **單一類型資料**
- 禁止同一變數在不同位置承載不同內容
- 每階段使用專用變數

### 原則四：事件驅動語意化

多個 if/else → 拆分為獨立事件處理函式：

```text
submitForm() 包含：驗證 + API 調用 + 結果處理
↓ 拆分為
validateData() → handleSuccess() / handleFailure()
```

### 原則五：可讀性優於簡潔性

價值排序：正確性 > 可讀性 > 可維護性 > 簡潔性

## 執行步驟

1. **命名檢查**：函式是否動詞開頭？變數是否無縮寫？
2. **長度檢查**：函式超過 10 行？→ 拆分
3. **職責檢查**：變數是否兼用？→ 分離
4. **事件檢查**：多個 if/else？→ 事件化
5. **可讀性測試**：5 分鐘內陌生人能理解？

## 檢查清單

### 命名
- [ ] 函式以動詞開頭
- [ ] 變數完整描述內容，無縮寫
- [ ] 類別說明業務職責

### 結構
- [ ] 每個函式 ≤ 10 行
- [ ] 無複合邏輯函式
- [ ] 變數職責單一

### 事件化
- [ ] if/else 已拆分為事件處理函式
- [ ] 函式名稱反映業務事件

### 驗證
- [ ] 陌生人 5 分鐘內能理解
- [ ] 翻譯成自然語言流暢
