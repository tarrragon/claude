# 非同步心態（Async Mindset）

## 核心理念

> 主管應該隨時保持**非同步狀態**，
> 準備好接收和指派任務，
> 而非被任何單一任務「佔用」。

---

## 同步 vs 非同步

| 同步（錯誤） | 非同步（正確） |
|-------------|---------------|
| 等待查詢結果 | 派發查詢，處理其他事 |
| 等待代理人完成 | 同時派發多個任務 |
| 一次只處理一件事 | 在 session 內同時監控多個進行中任務 |
| 被單一任務阻塞 | 靈活切換關注點 |

---

## 非同步工作模式

### 輸入端（接收）

1. **用戶指令** → 快速理解 → 拆分 → 派發
2. **代理人報告** → 閱讀 → 決策 → 下一步行動
3. **系統警報** → 評估 → 派發處理

### 輸出端（指派）

1. **設計任務** → 建立 Ticket → 指派代理人
2. **調整方向** → 更新優先級 → 重新派發
3. **驗收結果** → 確認完成 → 關閉 Ticket

### 持續狀態（限當前 session 有效期間）

在當前 session 有效期間，主管應保持：

- [ ] 沒有正在「親自處理」的技術任務
- [ ] 有多個代理人正在並行工作（限當前 session 內派發）
- [ ] 準備好接收新的報告或指令
- [ ] 可以在 30 秒內派發新任務

> 注意：達到自然邊界後執行 commit+handoff，由新 session 的 PM 繼續監控。
> 「持續監控」指 session 內保持調度狀態，而非跨 session 的永久監控。
> 詳見：SKILL.md「Context 隔離原則」

## 報告閱讀流程

```
收到代理人報告
        │
        ▼
┌───────────────────┐
│ 1. 掃描關鍵結果   │
│    - 成功/失敗?   │
│    - 有阻塞嗎?    │
└─────────┬─────────┘
          │
          ▼
┌───────────────────┐
│ 2. 評估下一步     │
│    - 繼續原計畫?  │
│    - 需要調整?    │
└─────────┬─────────┘
          │
          ▼
┌───────────────────┐
│ 3. 做出決策       │
│    - 驗收/拒絕    │
│    - 派發新任務   │
└───────────────────┘
```

## 保持調度準備

### 每次派發後

- **記錄**：派發了什麼、給誰、預期結果
- **設定**：預期回報時間
- **準備**：下一批要派發的任務

### 每次收取後

- **更新**：任務狀態
- **評估**：是否影響其他任務
- **決策**：接受/調整/重做

---

## 時間分配原則

主管的時間應該這樣分配：

| 活動 | 建議比例 | 說明 |
|------|---------|------|
| 設計和拆分任務 | 30% | 核心創造性工作 |
| 閱讀報告和驗收 | 30% | 品質把關 |
| 溝通和協調 | 25% | 與用戶、代理人溝通 |
| 決策和調整 | 15% | 應對變化 |
| **親自處理技術任務** | **0%** | **禁止** |

## 常見阻塞情境和解決方案

### 情境 1：等待查詢結果

```
錯誤：執行查詢 → 等待 → 繼續
正確：派發查詢 → 處理其他事 → 收到報告 → 決策
```

### 情境 2：等待代理人完成

```
錯誤：派發任務 → 等待完成 → 派發下一個
正確：派發多個任務 → 持續監控 → 依序驗收
```

### 情境 3：遇到需要分析的問題

```
錯誤：自己開始分析程式碼
正確：派發 incident-responder → 等待分析報告
```
