# 制式化內容生成原則（速查 stub）

> **完整方法論**：`.claude/methodologies/structured-content-generation-methodology.md`（按需讀取，含適用判準、設計模式、三層防護模型、與既有原則關係）。本檔僅保留核心原則與檢查清單。

有確定性 schema 的內容，結構由工具（CLI/函式）生成，人/agent 只提供語意值。Hook 檢查是保險（第 3 層），不是主防線。

## 核心原則

| 原則 | 說明 |
|------|------|
| 結構與語意分離 | 格式結構是確定性函式，應由工具承擔；語意值是創造性輸入，由人/agent 提供 |
| CLI 化優先 | 有 schema + 多寫入者 + 歷史格式錯誤 → 應封裝為 CLI 子命令 |
| 三層防護 | 第 1 層 CLI 制式化（源頭）→ 第 2 層 WARNING 含格式範例（即時引導）→ 第 3 層 Hook 事後檢查（保險） |

## 適用三條件（AND）

1. 有確定性 schema（欄位名、格式、排列順序可程式化定義）
2. 多個寫入者（2+ agent 或人 + agent）
3. 格式錯誤有歷史（hook 警告 / 手動修正觀察到）

三條件全滿足 → 應 CLI 化。僅 (1) 滿足 → 暫不（收益不足）。

## 何時讀完整方法論

| 情境 | 必讀 |
|------|------|
| 設計新 CLI 命令 / ticket body 章節 / 設定欄位 | 是（適用判準 + 設計模式 A/B） |
| 審查 hook 的 WARNING 訊息是否足夠 | 是（三層防護模型） |
| 判斷某寫入點是否需 CLI 化 | 是（適用三條件 + 不適用場景） |
| 日常開發、執行 Ticket | 否 |

## 檢查清單

設計新寫入點時：

- [ ] 有確定性 schema？+ 多寫入者？+ 歷史格式錯誤？
- [ ] 三條件全滿足 → 已規劃 CLI 子命令？
- [ ] CLI 化前過渡期 → WARNING 含正確格式範例？
- [ ] Hook 事後檢查已建立（保險層不因 CLI 化而移除）？
