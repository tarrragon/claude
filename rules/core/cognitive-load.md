# 認知負擔設計原則

本文件定義程式碼和任務設計中的認知負擔管理原則。

> **核心理念**：降低閱讀者的認知負擔是所有設計的終極目標。

---

## 核心概念

### 什麼是認知負擔？

認知負擔（Cognitive Load）是指閱讀者在理解程式碼時需要**同時記住**的資訊量。

> **Miller's Law**：人類工作記憶一次只能處理約 7±2 個項目。

### 設計目標

```
最小化外在認知負擔，讓閱讀者的認知資源用於
理解真正重要的事情（內在負擔）和建立心智模型（相關負擔）
```

## 量化標準

### 認知負擔指數

```
認知負擔指數 = 變數數 + 分支數 + 巢狀深度 + 依賴數
```

### 閾值表

| 指數 | 評估 | 行動 |
|------|------|------|
| 1-5 | 優良 | 維持 |
| 6-10 | 可接受 | 考慮優化 |
| 11-15 | 需重構 | 建立重構 Ticket |
| > 15 | 必須重構 | 立即處理 |

### 任務拆分閾值

| 條件 | 閾值 | 行動 |
|------|------|------|
| 需追蹤概念數 | > 7 個 | 必須拆分 |
| 需理解檔案數 | > 5 個 | 考慮拆分 |
| 跨架構層級數 | > 2 層 | 必須拆分 |
| 依賴模組數 | > 3 個 | 考慮拆分 |
| 修改檔案數 | > 5 個 | 必須拆分 |
| 目標檔案行數 | > 300 行 | 分析 domain 邊界，考慮拆分檔案 |
| 單一檔案承載 domain 數 | > 2 個 | 必須拆分（DDD 邊界違規） |
| 事件經過處理層數 | > 5 層 | 簡化流程或拆分為獨立處理鏈 |

> **回合耗盡 = 認知負擔過載的具體訊號**：代理人回合耗盡時，不只是「重試」或「PM 代做」，而是系統性訊號——某處的檔案體量或 domain 混合度需要降低。行數超標是症狀，domain 混合是病因。

## 3b 派發前閾值（PM 派發檢查門）

> **核心定位**：本章節是 PM 在 TDD Phase 3b（GREEN 實作派發）之前的整合性檢查門，將散佈在三份文件的閾值收斂為單頁三項核心指標，避免 PM 派發過大任務壓垮代理人認知負擔（3b 派發前閾值 ANA 收斂結論，具體 ticket 參工作日誌）。

### 三項核心閾值

| 閾值 | 拆分門檻 | 取得方式 |
|------|---------|---------|
| 功能職責數 | > 2 須拆分 | 數一數 ticket 涵蓋的獨立功能職責（一個職責 = 一個動詞 + 單一目標） |
| 修改檔案數 | > 5 須拆分 | `where.files` 欄位計數，或從 Problem Analysis 影響範圍推算 |
| Context Bundle tokens | > 3000 tokens 須拆分（軟上限）；> 5000 tokens 強制拆分 | claim 後 `ticket track show <id>` 查 PCB 區段字數，或 `wc -c` 估算 |

### 三明示

#### 閾值 1：功能職責數 > 2

**Why**：一個 ticket 涵蓋多重職責時，代理人需同時建立多套心智模型，工作記憶受 Miller's Law 7±2 限制，職責越多越易卡住或產出遺漏。

**Consequence**：派發後常見症狀為（1）代理人 commit 時遺漏部分職責；（2）回合耗盡前只完成一半；（3）跨職責的測試互相干擾，難以定位失敗根因。

**Action**：以「動詞 + 單一目標」描述每個職責；> 2 個動詞時依職責邊界拆分為多個 ticket，於 `spawned_tickets` 互相連結。

#### 閾值 2：修改檔案數 > 5

**Why**：跨檔案修改需代理人保持多檔案的內部一致性（命名、介面、行為），檔案越多一致性維護成本越高，且 Edit 工具的 old_string 唯一性檢查易因相似內容而失敗。

**Consequence**：> 5 檔修改的 ticket 常出現（1）部分檔案修改完整、其他檔案遺漏；（2）跨檔不一致導致測試失敗；（3）代理人需反覆讀取既有內容造成 token 浪費。

**Action**：依 domain 邊界拆分；同 domain 內 > 5 檔仍應評估「批量機械操作」（純重命名/格式化）vs「設計變更」分類，前者可用 inline script 一次處理（PC-069），後者必須拆 ticket。

#### 閾值 3：Context Bundle tokens > 3000

**Why**：Context Bundle 是 claim 時自動載入給代理人的前置脈絡，token 數越大代理人讀取成本越高，且 PCB 過大常代表 ticket 範圍過寬（牽涉多個 source ticket / 多個檔案的歷史脈絡）。

**Consequence**：PCB > 3000 tokens 的 ticket 常見（1）代理人讀完 PCB 已耗 30%+ context window；（2）PCB 中混雜不直接相關的歷史，代理人誤判任務焦點；（3）PCB > 5000 tokens 時代理人在實作前已逼近回合耗盡。

**Action**：> 3000 軟上限：審視 PCB 是否含與本 ticket acceptance 無關的歷史段落，可在 Problem Analysis 加註「PCB 部分段落僅供背景，本 ticket 焦點為 X」；> 5000 強制拆分：依 source ticket 數或檔案 domain 切分為多個 ticket，每個 ticket 限定 2-3 個 source。

### PM 派發前檢查清單

派發 3b 實作 ticket 前，PM 在主線程逐項勾選：

- [ ] 功能職責數 ≤ 2（一個 ticket 一個動詞 + 單一目標，最多兩個高度耦合的副職責）
- [ ] 修改檔案數 ≤ 5（`where.files` 或 Problem Analysis 影響範圍）
- [ ] Context Bundle tokens ≤ 3000（claim 後 `ticket track show` 查驗；> 3000 走豁免 / > 5000 必拆）
- [ ] 已對齊既有閾值（本檔 §量化標準/任務拆分閾值，無矛盾）
- [ ] 若任一項超標，已建立拆分 ticket 並更新 `spawned_tickets`

> **與 §量化標準/任務拆分閾值的關係**：上方 §量化標準 是通用閾值矩陣（適用程式碼 / 任務 / 文件設計多場景）；本章節是 **3b 派發前單頁檢查門**，從中精選三項最關鍵指標供 PM 派發決策即時對照。詳細閾值矩陣（含複雜度分級、並行派發判斷、程式碼品質）見 `@.claude/skills/cognitive-load-assessment/thresholds.md`。

### 跨進程同步修復豁免條款

> **理論依據**：跨進程同步修復豁免 ANA 收斂結論與其首例案例（具體 ticket 參工作日誌）。本條款是 §三項核心閾值「功能職責數 > 2 須拆分」的特例豁免，**不取代**通用閾值，僅在符合全部 5 特徵時放寬。

**適用情境**：當 IMP 滿足以下「跨進程同步修復」全部特徵時，可豁免「功能職責數 > 2 須拆分」閾值：

| 特徵 | 判定 |
|------|------|
| 跨進程數 | 2-3 個進程（如 CLI lib + Hook，或 validator + checker + linter） |
| 邏輯同步性 | 各進程實作同一根因修復，同步修法防止漂移（ARCH-020 教訓） |
| 修改檔案數 | ≤ 5（含測試檔，超過走通用閾值） |
| 預估執行時長 | ≤ 30 分鐘（PM 依過往案例估算，超過走通用閾值） |
| 根因數 | ≤ 2（多根因仍應拆 ANA → 多 IMP） |

**Why**：跨進程同步修復的拆分會破壞「同 bug 同 commit」一致性，導致兩 IMP 修法漂移風險（A 修了但 B 漏修，或 A 用方案 X、B 用方案 Y）；且 PM overhead（4 次 claim/complete）與測試重複設定成本高於合併執行。

**Consequence**：誤套用通用拆分閾值會：

1. 增加 ARCH-020 模式的修法漂移風險（首例已見：validator 漏修 hook 導致 3 天後 regression 重爆，同步修復才終結）
2. 拆 4 ticket 後 PM overhead = 4× claim + 4× complete + 4× commit + 4× 驗收
3. 測試難以共用 fixture（每 ticket 重建測試環境）

**Action**：豁免時必須於 IMP Problem Analysis 顯性記錄：

- 跨進程清單（哪些進程同步修復）
- 邏輯同步性說明（同根因 / 同修法）
- ARCH-020 或 PC-110 引用（理論依據）
- 預估執行時長（與基線案例對照）

**首筆基線案例**：28 分 37 秒 / 3 檔 / 12 測試 / 全綠 / 2 進程 / 2 根因（具體 ticket 參工作日誌）

**邊界澄清**：本豁免不適用於以下情境（仍走通用閾值）：

| 情境 | 處理方式 |
|------|---------|
| 跨進程 > 3 個 | 認知負擔超 7±2 上限，仍須拆分 |
| 跨 domain 修復（不同職責領域） | 合併會混淆責任，仍須拆分 |
| 多根因（> 2） | 先拆 ANA → 多 IMP，避免單 IMP 過載 |
| PM 估時 > 30 分鐘 | 超過經驗豁免上限，仍須拆分 |

> **後續觀測**：30 天觀測 ANA（具體 ticket 參工作日誌）將追蹤本豁免條款命中率與漂移率；若觀測發現命中後常超時，啟動豁免條款修訂。

## 自我檢查清單

### 函式層級

- [ ] 函式長度 <= 15 行（理想 5-10 行）
- [ ] 參數數量 <= 3
- [ ] 巢狀深度 <= 3
- [ ] 區域變數 < 5
- [ ] 函式名稱是動詞片語
- [ ] 函式只做一件事
- [ ] 無隱藏副作用

### 類別層級

- [ ] 類別方法數 <= 10
- [ ] 公開方法數 <= 5
- [ ] 類別只代表一個概念
- [ ] 類別名稱是名詞

### 命名層級

- [ ] 無縮寫或僅使用通用縮寫
- [ ] 變數名稱是名詞
- [ ] 函式名稱是動詞
- [ ] 布林變數是 is/has/can 開頭
- [ ] 不使用 data/info/value 等模糊詞

### 任務層級

- [ ] 能用「動詞 + 單一目標」描述
- [ ] 5 分鐘後還能記住任務目標
- [ ] 不需要同時記住超過 7 個概念

## 降低認知負擔的原則

### 原則 1：單一責任

> 一個函式只做一件事，讓讀者只需理解一個概念。

| 函式描述 | 責任數 | 評估 |
|---------|--------|------|
| 「計算總價」 | 1 | 符合 |
| 「計算總價並發送通知」 | 2 | 違反 |
| 「驗證、計算、儲存」 | 3 | 嚴重違反 |

### 原則 2：自說明命名

> 名稱本身就是文件，讓讀者不需要額外資訊。

| 命名 | 自說明程度 | 改善 |
|------|-----------|------|
| `calc()` | 低 | `calculateMonthlyRevenue()` |
| `isValid` | 中 | `isEmailFormatValid` |
| `processUserRegistration()` | 高 | 維持 |

### 原則 3：避免副作用

> 函式的行為應該可預測，讓讀者不需要追蹤隱藏狀態。

### 原則 4：扁平化結構

> 減少巢狀，讓讀者不需要追蹤深層上下文。

| 層級 | 評估 | 處理 |
|------|------|------|
| 1-2 | 優良 | 維持 |
| 3 | 可接受 | 考慮重構 |
| 4+ | 需重構 | 使用 Guard Clause 或提取函式 |

### 原則 5：資訊就近原則

> 相關的資訊應該放在一起，讓讀者不需要跳轉尋找。

## 與決策樹的關係

認知負擔是決策樹進行任務拆分的核心依據：

```
任務進入
    |
    v
評估認知負擔指數
    |
    +-- <= 10 --> 直接派發
    +-- > 10 --> 必須拆分後再派發
```
