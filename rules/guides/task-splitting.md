# 任務拆分指南

本文件定義基於認知負擔的任務拆分方法。

> 參考標準：[cognitive-load](../core/cognitive-load.md)

---

## 核心原則

> **任務拆分的目標：讓每個子任務的認知負擔在可管理範圍內**

人類工作記憶的容量約為 7 個項目。當一個任務需要同時追蹤超過這個數量的概念時，就應該拆分。

---

## 拆分觸發條件

### 任一條件符合即需拆分

| 條件 | 閾值 | 說明 |
|------|------|------|
| 變數狀態數 | > 5 個 | 單一任務需追蹤超過 5 個變數狀態 |
| 架構層級數 | > 2 層 | 任務跨越 3+ 架構層（如 UI + Controller + Domain） |
| 依賴關係數 | > 3 個 | 任務依賴超過 3 個其他模組 |
| 修改檔案數 | > 5 個 | 單一任務需修改超過 5 個檔案 |
| 認知負擔指數 | > 10 | 變數數 + 分支數 + 巢狀深度 + 依賴數 |

---

## 拆分策略

### 策略 1：按架構層拆分

**適用情境**：任務跨越多個 Clean Architecture 層級

```
原始任務：「實作新的書籍搜尋功能」
    |
    +-- 子任務 A：Domain 層 - 建立 SearchQuery 值物件
    +-- 子任務 B：Domain 層 - 建立 SearchResult Entity
    +-- 子任務 C：Application 層 - 建立 SearchBooksUseCase
    +-- 子任務 D：Infrastructure 層 - 實作 BookSearchRepository
    +-- 子任務 E：Presentation 層 - 建立搜尋 Widget

執行順序：A → B → C → D → E（由底層向上）
```

**原則**：
- 每個子任務只修改一個架構層
- 由底層（Domain）向上層（Presentation）實作
- 下層完成後才能開始上層

### 策略 2：按功能模組拆分

**適用情境**：任務涉及多個獨立模組

```
原始任務：「為所有 Repository 加入快取機制」
    |
    +-- 子任務 A：共用快取基礎設施（先執行）
    +-- 子任務 B：BookRepository 快取（可並行）
    +-- 子任務 C：AuthorRepository 快取（可並行）
    +-- 子任務 D：CategoryRepository 快取（可並行）

執行順序：A 先完成，B/C/D 可並行
```

**原則**：
- 共用模組先完成
- 獨立模組可並行
- 每個模組由同一個代理人完成

### 策略 3：按操作類型拆分

**適用情境**：任務包含不同類型的操作

```
原始任務：「重構命名並優化邏輯」
    |
    +-- 子任務 A：重命名操作（機械性，可並行）
    +-- 子任務 B：邏輯優化（需謹慎序列）

執行順序：A 全部可並行，B 序列執行
```

**原則**：
- 機械性操作（重命名、格式化）可大量並行
- 邏輯修改需要謹慎序列
- 不同類型不要混合在同一 Ticket

### 策略 4：按 TDD 階段拆分

**適用情境**：完整的功能開發

```
原始任務：「實作書籍評分功能」
    |
    +-- Phase 1 Ticket：功能設計（lavender）
    +-- Phase 2 Ticket：測試設計（sage）
    +-- Phase 3a Ticket：策略規劃（pepper）
    +-- Phase 3b Ticket：實作執行（parsley）
    +-- Phase 4 Ticket：重構優化（cinnamon）

執行順序：嚴格序列（Phase 順序）
```

## 拆分後檢查清單

### 複雜度檢查

- [ ] 認知負擔指數 < 10
- [ ] 變數狀態數 < 5
- [ ] 修改檔案數 < 5
- [ ] 架構層級數 <= 2
- [ ] 依賴模組數 <= 3

### 原子性檢查

- [ ] 能用「動詞 + 單一目標」描述
- [ ] 只有一個修改原因
- [ ] 所有驗收條件指向同一目標
- [ ] 可以獨立測試

### 依賴檢查

- [ ] 依賴關係明確標記
- [ ] 無循環依賴
- [ ] 前置任務已識別
- [ ] 可以獨立完成

## 拆分文件格式

```markdown
## 任務拆分計畫

### 原始任務
- **Ticket ID**: {原始 ID}
- **任務描述**: {描述}
- **複雜度評估**: {高}

### 拆分原因
- 認知負擔指數: {數值} > 10
- 跨架構層級: {數量} > 2

### 拆分策略
{策略名稱}：{策略描述}

### 子任務清單

| ID | 描述 | 代理人 | 依賴 | 可並行 |
|----|------|--------|------|--------|
| {ID}-A | {描述} | {代理人} | 無 | - |
| {ID}-B | {描述} | {代理人} | A | 否 |

### 驗收條件
- 所有子任務完成
- 整合測試通過
- 原始需求滿足
```

## 常見拆分錯誤

| 錯誤 | 症狀 | 解決 |
|------|------|------|
| 拆分過細 | 每個子任務只有幾行程式碼 | 確保每個子任務有足夠獨立性 |
| 拆分不完整 | 子任務之間有隱藏依賴 | 仔細分析依賴關係，明確標記 |
| 混合不同類型 | 同一 Ticket 包含重命名和邏輯修改 | 按操作類型分離 |
| 忽略測試 | 只拆分實作，沒有對應的測試 Ticket | 每個實作 Ticket 對應測試 Ticket |

## 拆分決策樹

```
任務進入
    |
    +-- 評估認知負擔指數
    |   +-- <= 10 --> 不需拆分，直接派發
    |   +-- > 10 --> 需要拆分
    |
    +-- 識別主要複雜度來源
    |   +-- 跨架構層 --> 按架構層拆分
    |   +-- 跨模組 --> 按功能模組拆分
    |   +-- 混合操作 --> 按操作類型拆分
    |   +-- 新功能 --> 按 TDD 階段拆分
    |
    +-- 產生子任務
    |
    +-- 驗證每個子任務
    |   +-- 認知負擔 < 10? --> 通過
    |   +-- 認知負擔 >= 10? --> 繼續拆分
    |
    +-- 標記依賴關係
    |
    +-- 建立 Ticket
```
