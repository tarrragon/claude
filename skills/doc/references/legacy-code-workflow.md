# Legacy Code 接手處理工作流程

本文件定義接手既有程式碼（Legacy Code）時的標準化六步驟流程。
適用於專案重啟、接手他人程式碼、或長期未維護的程式碼庫。

---

## 流程總覽

```
步驟 0: 編譯與建置驗證
    |
    v
步驟 1: Spec 標準化與功能實作盤點
    |
    v
步驟 2: 測試程式碼審查（非執行）
    |
    v
步驟 3: 提案與標準流程
    |
    v
步驟 4: 補充測試並逐 UC 驗證
    |
    v
步驟 5: 可觀測性審計
```

---

## 步驟 0：編譯與建置驗證

### 目的

確認專案能正常編譯和建置，排除基礎層級的阻礙。

### 操作流程

1. 執行安裝指令（如 `npm install`），確認依賴安裝無誤
2. 執行建置指令（如 `npm run build:dev`），觀察編譯錯誤
3. 修復所有編譯阻礙問題
4. 確認 build script 能正常完成

### 常見問題類型

| 問題類型 | 範例 | 處理方式 |
|---------|------|---------|
| 過時引用 | 引用已刪除/重命名的檔案 | 更新引用路徑或移除無效引用 |
| 模組宣告缺失 | ES module 缺少 `type: module` | 補充缺失的宣告 |
| Build script 引用過時檔名 | build.js 中的路徑與實際不符 | 更新 build script |
| 依賴版本衝突 | package-lock.json 過時 | 重新安裝依賴 |

### 經驗來源

- **v0.15.1** — content.js 缺失、ES module 宣告 `type: module` 缺失、build validation 引用過時檔名

### 檢查清單

- [ ] Build 成功完成
- [ ] 無過時的檔案引用
- [ ] Build 產物與 manifest 一致

## 步驟 1：Spec 標準化與功能實作盤點

### 目的

建立程式碼與規格文件的對應關係，釐清「已做了什麼」和「還缺什麼」。

### 操作流程

#### 分支 A：有 spec 文件時

1. 將 spec 格式標準化，對齊 `docs/spec/{domain}/` 目錄結構
2. 逐一比對 spec 中的功能描述與程式碼實作
3. 標記每個功能的狀態：已實作 / 部分實作 / 未實作
4. 盤點常數和配置的完整性（是否有缺失的常數定義）

#### 分支 B：無 spec 文件時

1. 閱讀程式碼，理解各模組的職責和功能
2. 依 domain 劃分，為每個 domain 產生規格草案
3. 產出 subdomain 層級的功能清單
4. 標記可確認的功能和待確認的功能

### 盤點範圍

| 盤點項目 | 說明 | 產出 |
|---------|------|------|
| 功能實作狀態 | 每個 spec 功能是否已實作 | 功能狀態清單 |
| 常數完整性 | ErrorCodes、STORAGE_KEYS 等是否齊全 | 缺失常數清單 |
| Domain 覆蓋 | 所有 domain 是否都有 spec | Domain 覆蓋率 |
| 程式碼與 spec 偏差 | 實作與 spec 不一致之處 | 偏差清單 |

### 經驗來源

- **v0.15.0** — ErrorCodes 系統分析（W1-001），盤點錯誤碼定義與實際使用的差異
- **v0.15.2** — STORAGE_KEYS 發現 5 個常數缺失，導致儲存操作使用硬編碼字串

### 檢查清單

- [ ] 所有 spec 已遷移至標準結構
- [ ] 每個功能有明確的實作狀態標記
- [ ] 常數定義完整（無硬編碼替代品）
- [ ] 產出功能盤點報告

## 步驟 2：測試程式碼審查（非執行）

### 目的

在不執行測試的前提下，審查測試程式碼的品質和覆蓋範圍，識別 skip 測試和過時測試。

> **重要**：本步驟是「審查」測試程式碼，不是「執行」測試。執行測試在步驟 4。

### 操作流程

1. 列出所有測試檔案，按 domain/usecase 分類
2. 審查測試是否符合 usecase 描述的驗收標準
3. 盤點所有被 skip 的測試（`test.skip`、`xit`、`xdescribe` 等）
4. 記錄每個 skip 測試的原因，評估是否仍然有效
5. 識別過時的測試（測試的功能已重構或移除）

### Skip 測試盤點格式

| 測試檔案 | 測試名稱 | Skip 原因 | 評估結果 |
|---------|---------|----------|---------|
| `xxx.test.js` | `should handle...` | 依賴未實作功能 | 待步驟 4 啟用 |
| `yyy.test.js` | `should validate...` | 功能已移除 | 可刪除 |

### 經驗來源

- **v0.16.1** — 191 個 skip 測試逐批啟用（191 -> 147 -> 65），分批處理避免一次處理過多失敗

### 檢查清單

- [ ] Skip 測試已列出並標記原因
- [ ] 過時測試已識別
- [ ] 測試與 usecase 的對應關係已釐清

---

## 步驟 3：提案與標準流程

### 目的

將 spec 中未實現的部分或新需求，透過標準提案流程轉為可執行的 Ticket。

### 操作流程

1. 彙整步驟 1 和步驟 2 的發現
2. 釐清 spec 中未實現的功能，列為候選提案
3. 向開發者確認接下來要處理的優先順序
4. 使用 `/doc` Skill 建立提案文件
5. 提案走三關式審查（詳見 `proposal-evaluation-guide.md`）
6. 審查通過後，產生 spec -> usecase -> ticket

### 經驗來源

- **v0.16.2** — 建立提案驅動需求追蹤機制（PROP-003~005、UseCase 遷移），確立從提案到 Ticket 的完整鏈路

### 檢查清單

- [ ] 步驟 1/2 的發現已彙整
- [ ] 提案已通過三關式審查
- [ ] spec 和 usecase 已建立
- [ ] Ticket 已開立

---

## 步驟 4：補充測試並逐 UC 驗證

### 目的

為缺少測試的程式碼補充基於 usecase 驗收標準的測試，並逐個 UC 執行驗證，採用「先局部再整體」策略。

### 操作流程

#### 4a. 補充測試

1. 確認 usecase 已存在（步驟 3 產出）
2. 依 usecase 的驗收標準（Acceptance Criteria）設計測試案例
3. 先建立 TDD Red Phase 測試（預期失敗）
4. 實作或修復程式碼使測試通過（Green Phase）
5. 評估是否需要重構（Refactor Phase）

#### 4b. 逐 UC 驗證

1. 選擇一個 UseCase 的測試檔案
2. 僅執行該 UC 相關的測試
3. 分析失敗原因，區分環境問題和程式碼問題
4. 修復後重新驗證
5. 確認通過後，進入下一個 UC
6. 所有 UC 通過後，再執行一次完整測試確認無交叉影響

### 失敗分類與處理

```
執行該 UC 測試
    |
    +-- 全部通過 --> 下一個 UC
    |
    +-- 部分失敗
        |
        +-- 環境問題（mock 配置、依賴缺失）--> 修復環境，重新執行
        +-- 程式碼問題（邏輯錯誤）--> 修復程式碼，重新執行
        +-- 測試本身問題（過時斷言）--> 更新測試，重新執行
```

### 環境差異警示

| 層級 | 通過的意義 | 不代表 |
|------|----------|--------|
| 單元測試 | 函式邏輯正確 | 模組間協作正確 |
| 整合測試（mock） | 介面契約正確 | 實際平台行為正確 |
| E2E 測試 | 流程端到端正確 | 所有邊界情況已覆蓋 |
| 實機測試 | 目標平台可運作 | 其他平台也可運作 |

> **經驗教訓**：mock 環境通過的測試不代表實機能正常運作。Chrome Extension 的 var hoisting、SPA 渲染等待、CSS 選擇器失效等問題只在實機才會出現。

### 經驗來源

- **v0.16.2** — W2-001~005 五個測試差距修復 Ticket，基於 usecase AC 補充測試
- **v0.15.0** — 面對 512 個測試失敗（87.6% 失敗率），大規模失敗造成無法有效分析
- **v0.15.3~v0.16.0** — 測試全過但實機發現 popup var hoisting、SPA 渲染等待、選擇器失效等完全不同的問題

### 檢查清單

- [ ] 每個 AC 至少有一個測試
- [ ] 每個 UC 獨立執行測試通過
- [ ] 失敗原因已分類（環境/程式碼/測試）
- [ ] 最終完整測試通過
- [ ] 已識別需要實機驗證的項目

## 步驟 5：可觀測性審計

### 目的

確保系統在出錯時能被發現，消除靜默失敗和盲區。

### 操作流程

1. 檢查所有關鍵流程是否有日誌輸出
2. 偵測靜默失敗（流程執行但無日誌輸出的路徑）
3. 檢查 error handler 是否正確註冊（時序問題）
4. 識別平台特有陷阱
5. 驗證錯誤能被正確捕獲和報告

### 審計項目

| 審計項目 | 檢查內容 | 風險等級 |
|---------|---------|---------|
| 靜默失敗 | try-catch 中只 return 不記錄 | 高 |
| Error handler 時序 | handler 是否在錯誤發生前已註冊 | 高 |
| 日誌覆蓋率 | 關鍵流程是否有啟動/完成/錯誤日誌 | 中 |
| 平台陷阱 | 變數提升、ASI 衝突、IIFE 問題 | 高 |
| 錯誤傳播 | 錯誤是否能從發生點傳播到使用者 | 中 |
| Promise 未 catch | unhandled rejection 導致異步靜默失敗 | 高 |
| async/await try-catch 遺漏 | await 呼叫未被 try-catch 包裹 | 高 |
| 回調中例外未捕獲 | setTimeout/setInterval 回調內的例外未處理 | 中 |

### 平台特有陷阱（Chrome Extension）

| 陷阱 | 說明 | 防護方式 |
|------|------|---------|
| var hoisting | 多 script 載入時變數衝突 | 使用 `let`/`const`，避免 `var` |
| IIFE/require ASI | 自動分號插入導致語法錯誤 | 明確加分號，使用 linter |
| SW error handler 延遲註冊 | Service Worker 啟動時錯誤未被捕獲 | 在最早的時機點註冊 handler |
| Content Script 隔離 | 注入頁面的 script 存取限制 | 使用 message passing |

### 經驗來源

- **v0.15.3** — popup 多 script 載入時變數衝突（var hoisting），導致 UI 無法正常顯示
- **v0.15.4** — 提取流程靜默失敗（無日誌輸出），問題無法被發現；Service Worker error handler 延遲註冊，早期錯誤被遺漏

### 檢查清單

- [ ] 無靜默失敗路徑（所有 catch 區塊有日誌）
- [ ] Error handler 在最早時機註冊
- [ ] 關鍵流程有啟動/完成/錯誤三階段日誌
- [ ] 平台特有陷阱已識別並加入防護
- [ ] 無 unhandled Promise rejection
- [ ] async/await 呼叫已有 try-catch 覆蓋
- [ ] setTimeout/setInterval 回調已有例外處理

## 步驟間的依賴關係

| 步驟 | 前置條件 | 產出物 | 後續步驟使用 |
|------|---------|--------|------------|
| 0 | 無 | 可編譯的專案 | 所有後續步驟 |
| 1 | 步驟 0 完成 | 功能盤點報告、標準化 spec | 步驟 2, 3 |
| 2 | 步驟 0 完成 | 測試審查報告、skip 清單 | 步驟 3, 4 |
| 3 | 步驟 1, 2 產出 | 提案、spec、usecase、ticket | 步驟 4 |
| 4 | 步驟 3 的 usecase | 測試程式碼、測試驗證結果 | 步驟 5 |
| 5 | 步驟 4 完成 | 可觀測性審計報告 | 維護階段 |

## 適用情境與排除情境

### 適用

| 情境 | 說明 |
|------|------|
| 專案重啟 | 長期暫停後重新啟動開發 |
| 接手他人程式碼 | 原作者已離開，需要接手維護 |
| 大版本升級前 | 升級前需要盤點現況 |
| 技術債清理 | 系統性處理累積的技術債務 |

### 排除

| 情境 | 說明 |
|------|------|
| 全新專案 | 從零開始的專案不需要 Legacy 流程 |
| 小幅修復 | 單一 bug fix 不需要完整六步驟 |
| 已有完整 spec 和測試的模組 | 直接走正常開發流程 |
