# 認知負擔量化標準

本文件定義認知負擔的量化閾值，作為決策樹、代理人、Code Review 的基本參考標準。

---

## 任務複雜度閾值

### 任務評估矩陣

| 指標 | 低複雜度 | 中複雜度 | 高複雜度（需拆分） |
|------|---------|---------|------------------|
| 變數狀態數 | 1-3 | 4-5 | > 5 |
| 修改檔案數 | 1-3 | 4-5 | > 5 |
| 架構層級數 | 1 | 2 | > 2 |
| 依賴模組數 | 0-1 | 2-3 | > 3 |
| 呼叫追蹤層 | 1-2 | 3 | > 3 |

### 複雜度等級定義

| 等級 | 定義 | 行動 |
|------|------|------|
| 低 | 所有指標都在「低複雜度」範圍 | 可直接執行 |
| 中 | 任一指標達到「中複雜度」 | 謹慎執行，考慮拆分 |
| 高 | 任一指標達到「高複雜度」 | 必須拆分後再執行 |

## 代理人升級閾值

### 強制升級條件

代理人遇到以下情況**必須升級到 PM**：

| 指標 | 閾值 | 判斷方法 | 行動 |
|------|------|---------|------|
| 需追蹤概念數 | > 7 | 數一數需要同時記住的概念 | 升級：任務需要拆分 |
| 需理解檔案數 | > 5 | 統計需要閱讀的檔案數 | 升級：可能跨越職責邊界 |
| 需追蹤呼叫層級 | > 3 | 追蹤最深的呼叫鏈 | 升級：架構複雜度過高 |
| 無法 5 分鐘理解 | 是 | 計時閱讀任務描述 | 升級：需要進一步分析 |

### 職責邊界問題

| 情況 | 識別方法 | 行動 |
|------|---------|------|
| 跨架構層 | 需同時修改 UI + Domain | 升級：需要多代理人協作 |
| 跨模組 | 需修改 3+ 模組 | 升級：影響範圍過大 |
| 不確定性 | 無法清晰描述任務目標 | 升級：需要進一步分析 |

## 並行派發判斷

### 並行安全條件

| 條件 | 可並行 | 需序列 |
|------|--------|--------|
| 架構層級 | 同一層 | 跨層 |
| 邏輯依賴 | 無 | 有 |
| 檔案重疊 | 無 | 有 |
| 操作類型 | 重命名/格式化 | 設計變更 |
| 測試衝突 | 無 | 有 |

### 大量並行適用情境

| 情境 | 說明 | 範例 |
|------|------|------|
| 同一層級修改 | 修改都在同一架構層 | 所有 Repository 檔案 |
| 純重命名/格式化 | 無邏輯變更 | 變數重命名跨數十個檔案 |
| 無邏輯依賴 | 各檔案獨立運作 | 獨立的 Widget 檔案 |
| 批量機械操作 | 規則明確的重複操作 | 導入路徑統一修正 |

## 認知負擔指數計算

### 簡易計算公式

```
認知負擔指數 = 變數數 + 分支數 + 巢狀深度 + 外部依賴數
```

### 評估標準

| 指數 | 等級 | 行動 |
|------|------|------|
| 1-5 | 低 | 維持 |
| 6-10 | 中 | 考慮重構 |
| 11-15 | 高 | 建議拆分 |
| > 15 | 極高 | 必須拆分 |

### 範例計算

```dart
// 範例函式
void processOrder(Order order) {
  // 變數數: 4 (order, items, total, discount)
  // 分支數: 3 (if-else-if)
  // 巢狀深度: 2
  // 外部依賴數: 2 (OrderRepository, PricingService)

  // 認知負擔指數 = 4 + 3 + 2 + 2 = 11 (高)
  // 建議：拆分為更小的函式
}
```

## 拆分觸發條件

### 任一條件觸發即需拆分

| 條件 | 閾值 | 說明 |
|------|------|------|
| 變數狀態數 | > 5 | 單一任務需追蹤超過 5 個變數狀態 |
| 架構層級數 | > 2 | 任務跨越 3+ 架構層 |
| 依賴關係數 | > 3 | 任務依賴超過 3 個其他模組 |
| 修改檔案數 | > 5 | 單一任務需修改超過 5 個檔案 |
| 認知負擔指數 | > 10 | 計算結果超過閾值 |

### 拆分後檢查

每個子任務應滿足：

- [ ] 認知負擔指數 < 10
- [ ] 變數狀態數 < 5
- [ ] 修改檔案數 < 5
- [ ] 能用「動詞 + 單一目標」描述
- [ ] 不依賴其他未完成的子任務（或明確標記依賴）

## 使用指南

### PM 使用場景

1. **任務分派前**：評估任務複雜度，決定是否需要拆分
2. **並行派發時**：檢查並行安全條件
3. **進度追蹤時**：識別卡住的任務是否因複雜度過高

### 代理人使用場景

1. **任務開始前**：自我評估是否能處理
2. **執行中發現困難**：判斷是否需要升級
3. **Code Review 時**：評估程式碼品質

### Code Review 使用場景

1. **函式評估**：檢查長度、參數、巢狀
2. **類別評估**：檢查方法數、依賴數
3. **命名評估**：檢查清晰度和一致性
