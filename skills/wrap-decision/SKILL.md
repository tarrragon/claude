# WRAP 決策框架 Skill

**版本**: v1.0.0
**建立日期**: 2026-04-10
**狀態**: 穩定
**來源**: 《零偏見決斷法》(Decisive) — Chip Heath & Dan Heath + PC-051 實戰教訓

## 概述

WRAP 是一套系統化的決策引導框架，防止 PM 在自動駕駛模式中做出低品質決策。

核心理念：**提醒 PM「你是有選擇的」，而不是告訴 PM 正確答案。**

框架結構：錨點確認 → W（擴增選項）→ R（實境檢驗）→ A（拉開距離）→ P（準備好犯錯）→ 絆腳索（持續監控）

## 觸發條件

| 情境 | 識別特徵 | 模式 |
|------|---------|------|
| 連續失敗 | 同一問題修改 2+ 次仍失敗 | 快速 |
| 被困住 | PM 表達「做不到」「沒辦法」「CLI 不支援」 | 快速 |
| 偏離核心 | 連續 2+ 個 Ticket 不在核心 Wave | 完整 |
| 重大決策 | 影響架構/流程/版本方向的非技術決策 | 完整 |
| 救火排擠 | 花 > 30 分鐘在非原定計畫的問題上 | 快速 |
| **ANA Ticket** | 任何 ANA 類型 Ticket 的分析過程 | 快速+ |
| **Debug/修復** | 錯誤根因調查、代理人失敗歸因、測試失敗分析 | 快速 |
| **提案評估** | 評估 proposals/ 中的提案可行性 | 完整 |

> **0.18.0-W4-002 教訓**：未用 WRAP 時 PM 將代理人失敗歸因於能力不足（錯誤），用 WRAP 後系統性發現 3 層問題：Hook JSON 格式、觸發頻率、PM 判斷偏誤。分析/debug 場景的決策深度不足是反覆出現的模式，WRAP 的多選項擴增和實境檢驗能有效防止衝動歸因。

**快速模式**（5 分鐘）：錨點 + W + 基本率問題（R 核心）+ 機會成本（A 核心）+ 決定
**完整模式**（15-30 分鐘）：全 6 階段

## 核心流程

### 0. 錨點確認（Anchor Check）

在進入任何分析前，先回答兩組問題：

**第一錨點（產品決策）**：

| 問題 | 目的 |
|------|------|
| 「誰是我們的客戶？」 | 建立決策錨點（本專案：使用 Readmoo 的讀者） |
| 「當前版本的核心目標是什麼？」 | 對齊里程碑（查 CLAUDE.md 里程碑章節） |

**第二錨點（流程決策）**：

| 問題 | 目的 |
|------|------|
| 「這個流程改善影響 PM 決策品質或開發效率嗎？」 | 區分「改善工具」vs「低價值維護」 |
| 「不做這個改善，會重複付出什麼代價？」 | 評估一次性投入 vs 持續回報 |

**閘門判斷**：

```
這個問題影響核心客戶/核心目標嗎？
    +-- 是 → 進入 WRAP 流程
    +-- 否 → 這個流程改善影響 PM 決策品質或開發效率嗎？
    |         +-- 是且持續回報 > 一次性投入 → 進入 WRAP 流程
    |         +-- 是但低回報 → 建提案（docs/proposals/）→ 回到核心任務
    |         +-- 否 → 記錄 todolist → 回到核心任務
    +-- 純粹救火 → 建 Ticket + P2 → 回到核心任務
```

> 引用：英特普拉思特「病人才是我們的客戶」— 確立後，所有困難的決定都有了錨點。

---

### W — Widen Your Options（擴增選項）

**核心問句**：「還有什麼其他方式可以達成目標？」

#### 爬梯子法（由近到遠）

| 層級 | 搜尋範圍 | 做法 |
|------|---------|------|
| 0. 身邊 | 本專案既有程式碼/模組 | grep / LSP 搜尋既有 API |
| 1. 同層 | 本專案其他模組的做法 | 檢查類似功能的實作 |
| 2. 同領域 | 社群（GitHub Issues、文件） | WebSearch 搜尋 |
| 3. 跨領域 | 其他工具/框架的類比方案 | AI 內建知識調用 |

每層找到可行方案就停止，找不到才往上爬。

#### 品質檢查

| 檢查 | 問題 | 不通過的信號 |
|------|------|-------------|
| **選項消失測試** | 如果以上全部不能選，還能怎麼做？ | 想不出來 → 框架太窄 |
| **假選項偵測** | 選項之間有實質差異嗎？ | 全部指向同一根因/同一途徑 → 季辛吉陷阱 |
| **框架測試** | 能否用一句話概括所有選項？ | 可以 → 選項多元性不足 |

> 品質檢查不通過 → 強制回到爬梯子重新擴增。

---

### R — Reality-Test Your Assumptions（實境檢驗）

**核心問句**：「需要什麼事證才能證明這個方法可行？」

#### 基本率 > 預測

| 壞問題（預測） | 好問題（基本率） |
|---------------|----------------|
| 「這樣改會成功嗎？」 | 「這類問題的常見解法有哪些？成功率？」 |
| 「這是不是根因？」 | 「這類症狀的常見根因分佈是什麼？」 |
| 「代理人會完成嗎？」 | 「代理人回合耗盡的常見閾值是什麼？」 |

#### Zoom Out + Zoom In

| 方法 | 做法 | 產出 |
|------|------|------|
| Zoom Out | 搜尋基本率：多少人遇到？常見解法？ | 統計性判斷 |
| Zoom In | 讀具體案例內文、git diff 實際變更 | 直覺性判斷 |

兩者缺一不可：基本率建立基準，近距離檢視觸動直覺。

#### 試水溫（Ooch）

對每個選項問：「有沒有辦法用最小成本驗證？能在 5 分鐘內得到初步答案嗎？」
- 能 → 先試水溫再決策
- 不能 → 進入完整分析

#### 鋼鐵人測試（Steelman Test）

選定方案後、執行前：
1. 「用最有力的方式陳述被你放棄的選項的優點」
2. 「列出你選定方案的三個缺點」
- 做得到 → 決策品質足夠
- 做不到 → 回到 R 階段補充

---

### A — Attain Distance（拉開距離）

**核心問句**：「投入這個問題的時間，會擠壓哪個更重要的目標？」

#### 機會成本顯性化

每個選項旁邊強制列出：

```
選項 A：[做法]
  機會成本：花 X 時間，這段時間可以 [替代用途]
  風險：[可能失敗的條件]
  不選其他的代價：[放棄什麼]
```

> 引用：DVD 實驗 — 只加上「不買，留下 14.99 美元買別的」，不買率從 25% 升到 45%。

#### 核心優先事項對齊

| 問題 | 目的 |
|------|------|
| 「這個決策服務於哪個優先事項？」 | 對齊 |
| 「如果優先事項衝突，哪個排第一？」 | 排序 |
| 「投入這個問題的時間，會擠壓哪個更重要的目標？」 | 機會成本 |

排序確定後，低優先事項的決策自動有了答案 — 記錄 Ticket 延後，不投入時間。

#### 提案系統對接

不是「待辦清單」（焦慮全部做完），而是「安全停放區」（想法被記錄，明確標示「現在不做」）。

| 系統 | 角色 |
|------|------|
| `docs/proposals/` | 安全停放區 — 記錄想法，不代表要做 |
| Ticket 系統（pending） | 承諾清單 — 已決定要做，排入排程 |
| 版本里程碑（CLAUDE.md） | 核心優先事項 — 這個版本的目標 |

---

### P — Prepare to Be Wrong（準備好犯錯）

**核心問句**：「假設這個方案失敗了，最可能的原因是什麼？」

決策確定後、執行前，完成四項檢查：

| 檢查 | 問題 | 判斷 |
|------|------|------|
| **行前預想** | 假設 12 小時後失敗了，列出 3 個最可能原因 | 任一原因可能性 > 50% → 重新評估 |
| **未來區間** | 最好的結果？最壞的結果？ | 兩端都能接受嗎？ |
| **安全係數** | 預估時間/資源 x 1.3-1.5 | 緩衝夠嗎？ |
| **回退計畫** | 失敗怎麼回退？回退成本？ | 回退成本 > 收益 → 重新評估 |

---

### 5. 絆腳索（Tripwire）

**核心理念**：絆腳索不保證做出正確決定，但至少讓你意識到「是該做決定的時候了」。

#### 絆腳索類型

| 類型 | 觸發條件 | 動作 |
|------|---------|------|
| 期限型 | 非核心問題花 > 15 分鐘 | 建 Ticket 延後，回到核心 |
| 失敗型 | 同一修改連續 2 次失敗 | 搜尋社群或換方向 |
| 偏離型 | 連續 2+ 個 Ticket 不在核心 Wave | 回到核心任務 |
| 回退型 | 已回退過一次 | 完全停止，換方向 |
| 嘗試型 | 同一問題修改嘗試 2 次 | 搜尋社群 / 向外求解 |

#### 正面絆腳索

不只防止壞事，也捕捉好事：
- Ticket 完成後問：「有什麼意外的成功值得記錄？」
- 對應 `/continuous-learning` Skill

#### 切割 = Checkpoint

WRAP 每階段之間是切割點 — 強迫問「要不要繼續」：
- W 完成 → 「選項品質夠嗎？」
- R 完成 → 「證據支持哪個選項？」
- A 完成 → 「這符合核心優先事項嗎？」
- P 完成 → 「行前預想的風險可接受嗎？」
