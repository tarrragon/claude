# Dimension：State Storage（正式狀態與資料儲存）

本維度處理「不可遺失的正式狀態放哪裡、怎麼保護」。任何 SaaS 都有帳號、訂閱、業務資料這類 source of truth、所以本維度必展開。選型的第一個動作是把資料分成「正式狀態」與「衍生資料」兩堆 — 正式狀態遺失等於事故、衍生資料（快取、索引、報表）可以從正式狀態重建；兩堆的儲存策略、備份壓力、選型自由度完全不同。

---

## 訪談問題

| 問題                                                                                            | 為什麼問                                                                                                                                                                       |
| ----------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 核心資料長什麼形狀 — 關聯密集（訂單連客戶連商品）、文件式（表單 / 設定 blob）、還是 key-value？ | 資料形狀是 DB 類型的主要依據、比「哪個 DB 流行」可靠                                                                                                                           |
| 有沒有跨多筆資料要嘛全成功要嘛全失敗的操作（扣款 + 建訂單）？                                   | 有 → transaction 邊界是硬需求、relational DB 的核心優勢；邊界沿 domain 配置 — 同 domain 內的寫入同 transaction、跨 domain 的一致性走 event + outbox、不開跨 domain transaction |
| 檔案類資料（上傳圖片、報表、附件）量級？                                                        | 檔案進 object storage、進 DB 是常見的早期錯誤（備份爆量、查詢變慢）                                                                                                            |
| 需要全文搜尋嗎？day one 還是之後？                                                              | DB 內建搜尋能撐很久、獨立搜尋引擎是「觸發才引入」的元件                                                                                                                        |
| 多租戶的話、租戶資料要隔離到什麼層級？                                                          | 決定 shared schema / schema-per-tenant / db-per-tenant、事後改是大型遷移                                                                                                       |

**反向問**：「哪一張表的資料弄丟了、公司會出事？」— 答案圈出備份與還原驗證的優先範圍；答不出來表示正式狀態 / 衍生資料的分類還沒做。

## 候選類型差異

| 類型                             | 適合                                    | 代價                                                    |
| -------------------------------- | --------------------------------------- | ------------------------------------------------------- |
| Relational（PostgreSQL / MySQL） | 關聯密集、需要 transaction、schema 明確 | schema 變更要走 migration 紀律                          |
| Document（MongoDB 類）           | schema 高頻變動、文件自包含             | 跨文件一致性弱、關聯查詢成本高、事後補 transaction 困難 |
| Key-value（DynamoDB 類）         | 存取模式固定且已知、規模極大            | 查詢模式要先於 schema 設計、ad-hoc 查詢幾乎不可行       |
| Embedded（SQLite）               | 單機工具、邊緣場景                      | 多寫入者與水平擴展受限                                  |
| Object storage（S3 類）          | 檔案、備份、報表產物                    | 沒有查詢能力、要配 metadata 索引                        |

**SaaS day one 預設**：managed relational DB + object storage。理由：MVP 階段存取模式未知、relational 的 ad-hoc 查詢自由度是「需求還會變」的保險；managed 把備份、升級、failover 外包給平台、符合小團隊定錨。使用者點名 document / KV 時、回到資料形狀與查詢模式確認 — 確認成立就採納、選型沒有標準答案、只有跟需求對齊的答案。

**外包深度與 bundle 判讀**：資料庫常是整個 bundle 決策的錨點 —— 因為它是跨能力 bundle 的核心那一塊（見 `principles/capability-outsourcing-depth.md`）。先分清「managed」的深度差：純 managed 關聯式 DB（RDS / Cloud SQL / Neon 類、只外包 DB 維運、schema 與 query 仍是你的）跟跨能力 bundle（同時給 DB + 認證 + 物件儲存 + 即時推送的後端平台）不在同一層。資料庫一旦選了 bundle、就要回頭看 security（認證用不用它的）、cache、async-queue（即時推送）哪幾塊一起用它、把這幾塊收進同一筆 bundle 決策、而不是各維度各接一個獨立服務。逐塊判「用它值不值得換整包遷出代價」、bundle 的整包遷出代價記進 tripwire。

**多租戶資料模型**：預設 shared schema + 每表 `tenant_id` + 查詢層強制 scope（ORM default scope 或 row-level security）。schema-per-tenant 與 db-per-tenant 在「合規要求實體隔離」或「大客戶要獨立備份 / 還原」時才引入、代價是 migration 與維運成本乘上租戶數。

## 防護底線（non-negotiable）

1. **Migration 版本化**：schema 變更全部走 migration 檔、進 repo、禁手改 production。這是部署可重現與回滾的前提。
2. **自動備份 + 還原驗證**：託管平台開自動備份只是一半、上線前實際還原一次到隔離環境才算數。還原步驟寫進 README。
3. **租戶隔離在查詢層強制**：多租戶下、依賴「工程師記得加 where tenant_id」是事故時間表、不是控制。用 ORM scope 或 RLS 讓漏寫變成查不到而不是查到別人。
4. **連線池上限明確設定**：池上限 × 實例數 ≤ DB 連線上限。預設值往往在加第二台實例時打爆 DB、day one 設好是零成本。
5. **慢查詢紀錄開啟**：規模成長「資料庫變慢」撞牆訊號的偵測源、設定即完成。

## 延後與 tripwire

| 可延後項      | 建議重評條件                                              |
| ------------- | --------------------------------------------------------- |
| Read replica  | 讀寫比明顯失衡（10:1 為起始門檻、依定錨調整）且讀延遲惡化 |
| 獨立搜尋引擎  | DB 全文搜尋的查詢延遲或功能（斷詞、權重）撐不住           |
| 分片 / 分區   | 單表行數或單庫容量接近平台建議上限                        |
| db-per-tenant | 合約要求實體隔離、或單一大客戶資料量壓過其他全部          |
