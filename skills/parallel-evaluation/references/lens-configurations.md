# 視角配置詳細

各情境的完整視角定義和檢查項目。

> Agent 提示模板: lens-prompts.md

## 情境 A: 程式碼變更審查

### Reuse 視角

**Agent 類型**: Explore
**掃描焦點**: 重複功能、可用既有 utility

**檢查項目**:

- 變更中是否有可以用既有 utility 替代的程式碼
- 是否引入了與現有功能重複的新函式
- 是否有跨模組的 copy-paste 程式碼
- 新的 helper/utility 是否已存在於其他模組

### Quality 視角

**Agent 類型**: feature-dev:code-reviewer
**掃描焦點**: 冗餘狀態、copy-paste、leaky abstraction

**檢查項目**:

- 冗餘的狀態管理（多處追蹤同一個值）
- 抽象洩漏（上層暴露了不該暴露的底層細節）
- 不必要的間接層（wrapper 只是轉發，無附加邏輯）
- 命名與實際行為不一致

### Efficiency 視角

**Agent 類型**: feature-dev:code-reviewer
**掃描焦點**: 不必要的工作、missed concurrency、memory leak

**檢查項目**:

- 重複計算（可快取或提前計算的值）
- 錯失的並行機會（可同時執行的獨立操作）
- 記憶體洩漏風險（未關閉的 Stream、未取消的 Subscription）
- 不必要的 I/O 操作

## 情境 B: 重構決策評估

### Redundancy 視角

**Agent 類型**: Explore
**掃描焦點**: 重複設計、冗餘模組、功能重疊

**檢查項目**:

- 跨模組的功能重疊（兩個模組做類似的事）
- 重複的資料轉換邏輯
- 冗餘的中間層或 adapter
- 可合併的相似介面

### Coupling 視角

**Agent 類型**: Explore
**掃描焦點**: 不必要的依賴、耦合度、變更擴散範圍

**檢查項目**:

- 跨層依賴（Presentation 直接依賴 Infrastructure）
- 循環依賴
- 高扇出（一個模組依賴過多其他模組）
- 變更一處需要修改多處的跡象

### Complexity 視角

**Agent 類型**: Explore
**掃描焦點**: 認知負擔熱點、過度工程、可簡化的抽象

**檢查項目**:

- 認知負擔指數 > 10 的函式或類別
- 過度工程（為假設的未來需求設計的抽象）
- 深層繼承鏈或過多的介面層
- 可用簡單方案替代的複雜實作

## 情境 C: 架構方案評估

### Consistency 視角

**Agent 類型**: Explore
**掃描焦點**: 與現有架構模式是否一致

**檢查項目**:

- 是否遵循現有的分層架構（Domain → UseCase → Presentation）
- 命名慣例是否與既有程式碼一致
- 錯誤處理模式是否與既有模式一致
- 資料流方向是否正確

### Impact 視角

**Agent 類型**: Explore
**掃描焦點**: 變更範圍、影響模組數、遷移成本

**檢查項目**:

- 直接影響的檔案/模組數量
- 間接影響（依賴此模組的其他模組）
- 需要修改的測試數量
- 遷移期間的向後相容性需求

### Simplicity 視角

**Agent 類型**: Plan
**掃描焦點**: 是否存在更簡單的替代方案

**檢查項目**:

- 是否可以用現有機制解決（不需新架構）
- 是否過度設計（用更少的抽象能否達成）
- 是否有業界已驗證的更簡單方案

---

## 情境 D: 功能設計評估

### Overlap 視角

**Agent 類型**: Explore
**掃描焦點**: 與現有功能是否重疊

**檢查項目**:

- 新功能是否與既有功能有 > 50% 的行為重疊
- 是否可以擴展既有功能而非新建
- 是否有類似的 Domain Entity 或 Value Object 已存在

### Fit 視角

**Agent 類型**: Explore
**掃描焦點**: 是否符合現有 Domain Model 和資料流

**檢查項目**:

- 新功能的 Entity 是否自然融入現有 Domain Model
- 資料流是否需要新的轉換層
- 是否需要修改現有的核心介面

### Scope 視角

**Agent 類型**: Plan
**掃描焦點**: 範圍是否過大、是否可分階段交付

**檢查項目**:

- 認知負擔指數估算
- 是否可拆為 MVP + 增量
- 是否有隱含的前置依賴

## 情境 E: 系統冗餘偵測

### Code Duplication 視角

**Agent 類型**: Explore
**掃描焦點**: 跨模組的重複實作

**檢查項目**:

- 跨目錄的相似函式（功能相同但實作不同）
- 重複的資料驗證邏輯
- 相似的錯誤處理模式

### State Redundancy 視角

**Agent 類型**: Explore
**掃描焦點**: 冗餘的狀態管理、重複快取

**檢查項目**:

- 多處追蹤同一個資料的不同副本
- 重複的快取策略
- 可合併的 Provider/ViewModel

### Interface Bloat 視角

**Agent 類型**: Explore
**掃描焦點**: 過大的介面、未使用的 API

**檢查項目**:

- 公開方法數 > 10 的類別
- 無引用者的公開方法
- 過度泛化的介面（參數或方法數過多）

## 情境 F: 結論/報告審查

### Evidence 視角

**Agent 類型**: Explore
**掃描焦點**: 結論是否有充分的事實/程式碼佐證

**檢查項目**:

- 每個結論是否引用了具體的程式碼位置或錯誤訊息
- 是否有推測性結論未標註為推測
- 因果關係是否有直接證據支持

### Alternatives 視角

**Agent 類型**: Plan
**掃描焦點**: 是否遺漏其他可能的根因或解法

**檢查項目**:

- 報告是否只考慮了一個根因
- 是否有其他可能的解釋
- 是否有更低風險或更低成本的解法

### Scope 視角

**Agent 類型**: Explore
**掃描焦點**: 結論的影響範圍是否被低估或高估

**檢查項目**:

- 修復建議是否考慮了所有引用者
- 影響範圍估計是否包含間接依賴
- 是否遺漏了其他可能受影響的模組

---

## 情境 G: 系統設計評估

### Consistency 視角

**Agent 類型**: Explore
**掃描焦點**: 交叉引用完整性、術語一致性、無矛盾

**檢查項目**:

- 所有引用是否雙向（A 引用 B 時，B 也引用 A）
- 術語是否全域一致（同一概念不同名稱）
- 不同文件中的相同規則是否有矛盾
- 相關文件列表是否完整且準確

### Completeness 視角

**Agent 類型**: Explore
**掃描焦點**: 覆蓋範圍缺口、缺少的段落、單向引用

**檢查項目**:

- 是否有情境/場景未被覆蓋（缺口分析）
- 必要的段落是否都存在（如「相關文件」「適用時機」）
- 正交工具之間是否有區別說明
- 新增內容是否已在所有相關入口點可被發現

### Cognitive Load 視角

**Agent 類型**: Explore
**掃描焦點**: 文件可讀性、資訊密度、Progressive Disclosure

**檢查項目**:

- 主文件是否控制在 5,000 字以內（Progressive Disclosure）
- 資訊是否按需分層（核心 → 詳細 → 參考）
- 表格和清單是否易於快速掃描
- 是否有過多重複內容（應引用而非複製）

## 常駐委員：linux（Good Taste 品質把關）

> 所有情境自動加入，不限於特定情境。

**Agent 類型**: linux
**掃描焦點**: 設計簡潔性、問題真實性、特例消除、向後相容

**檢查項目**:

- 設計是否過度複雜？能用更簡單的方式達成嗎？
- 是否在解決真實問題？還是假想的未來需求？
- 是否有不必要的特例？能透過重新設計消除嗎？
- 變更是否破壞任何現有行為或使用者體驗？

**與 Worth-It Filter 的整合**：

linux 評分直接對應 Worth-It Filter 的改善幅度，走同一套規則，無特例：

| linux 評分 | 對應 Worth-It Filter | 行為 |
|-----------|---------------------|------|
| Garbage | 高幅度發現 | 依 Worth-It Filter 決策（高幅度 = 必須/值得執行） |
| Acceptable | 中幅度發現 | 依 Worth-It Filter 決策（中幅度 + 風險判斷） |
| Good taste | 無發現 | 無需行動 |
