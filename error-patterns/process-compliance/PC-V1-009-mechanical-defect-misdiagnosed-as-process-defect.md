## 症狀

- 一個**機械性、確定性可測**的缺陷（import 殘留 / NameError / 型別不符 / broken link / 語法錯）造成事故。
- 事後歸因得出「缺少**人工審查關卡**」的結論，據此提議加一道人工流程（PR review / 人工複查 / 簽核）。
- 提議的人工關卡實際上**攔不住該缺陷**——它要求人腦做 def-use 分析或全域一致性檢查，正是人類最不擅長、機器最擅長的事。
- 加上的流程引入儀式成本與「審查過了所以安全」的假信心，並把零成本的自動化方案擠出設計空間。

## 根因分析

### 行為模式

事故發生時，把「沒有被攔下」直覺歸因為「沒有人去看」，於是補一個「人去看」的關卡。但缺陷的型態是**確定性可判定**的（執行一次 import / linter / test 即得二元答案），其攔截工具應是自動檢查而非人工判斷。誤診的本質是把「機械完整性問題」錯置到「需要人類判斷的設計問題」這個範疇。

### 結構性原因

1. **歸因停在「缺關卡」層級**，未追問「該缺陷是機械可測還是需人類判斷」。
2. **人工 review 被過度泛化為萬用攔截手段**，忽略它對機械缺陷的攔截率極低（reviewer 需在腦中重建 def-use / 全域一致性，易漏）。
3. **忽略爆炸半徑與攔截點是兩個問題**：人工關卡即使攔到也只在單點，對「已傳染全體」的放大器無作用。

### 具體案例（說明用）

某框架以 overlay sync 直推所有 consumer。一次不完整重構留下 import 殘留（`NameError`），零審查傳染全部 consumer 致崩潰。事後提議「sync-push 改 branch+PR 加人工 review 閘門」。多視角審查指出：人工 reviewer 盯著「移除 import X」的 diff，需對整個模組做 def-use 分析才能發現他處仍用 X——攔不住。正解是 push 前跑 import smoke test（一行、0.1 秒、二元結果、零誤報）消除整個缺陷類別；爆炸半徑另由下游版本 pin 處理。人工 PR 對機械缺陷過重、對爆炸半徑無效，且在單人 repo（reviewer=author）必退化成自 merge 空殼。

## 解決方案

### 判斷準則（決策時自問）

| 問題 | 若「是」→ |
|------|----------|
| 此缺陷能否用一次自動執行（import / lint / test）得二元判定？ | 用**自動閘門**消除整個缺陷類別，不加人工關卡 |
| 此關卡要求人腦做 def-use / 全域一致性分析？ | 該工作屬機器強項，人工攔截率低，改自動化 |
| 事故痛點是「缺陷產生」還是「缺陷瞬時傳染全體」？ | 前者用完整性閘門、後者用版本 pin / 隔離，兩者分開解 |
| 此關卡在實際組織下 reviewer 是否 ≠ author？ | 否則 review gate 退化空殼，無實質價值 |

### Good Taste 原則

面對特殊情況，正確動作是**用資料結構 / 自動化把它消除成常規情況**，而非加一道人工關卡去攔它。一個會被繞過的 gate 比沒有 gate 更糟——它製造合規假象，還把乾淨的自動化擠出設計空間。

## 預防措施

| 措施 | 說明 |
|------|------|
| **歸因先分型** | 事故歸因時先判「機械可測 vs 需人類判斷」，再決定攔截工具是自動還是人工 |
| **機械缺陷一律自動閘門** | import smoke test / linter / test 作 push/CI 前 fail-fast，不依賴人工 review |
| **閘門速度準則** | 閘門必須快到沒人想繞（push 前秒級自檢），而非慢到逼人繞（人工 PR） |
