# PC-BAL-004: 派發者將未查證的事實寫入 ticket context，成為執行者的錯誤前提

## 症狀

派發者為 ticket 撰寫 Context Bundle 時，把「快速掃描得到的印象」當作已驗證事實寫入。執行者依該前提開工，在實作中發現前提不成立，需先推翻再重做判斷。

前提錯誤有兩種代價：執行者花費回合修正而非產出，以及若執行者未察覺而照單全收，錯誤前提會固化進實作與後續文件。

典型的錯誤前提形態：

| 形態 | 例 |
|------|-----|
| 斷言某物不存在（未查） | 「該目錄下無既有 README，由你判斷落點」——README 自初始 commit 即存在 |
| 斷言某設定缺失（未查） | 「評估能否以依賴宣告取代旗標」——宣告一直都在 |
| 數量來自不精確的搜尋 | 「4 個檔案含 PEP 723」——未錨定 grep 命中字串常數，實為 3 |
| 判準基於片面理解 | 「dev group 子集」——實際可用集合為 `[project].dependencies` 與 dev group 的聯集 |

## 根因

派發者為降低執行者的探索成本而預先蒐集 context，此動機正確（PC-040 要求 context 入 ticket 而非 prompt）。缺失在於蒐集與查證被視為同一動作。

**機制**：`grep -c`、`ls`、`find` 的輸出是「搜尋結果」，不是「事實」。搜尋的精確度取決於 pattern 設計——未錨定的子字串比對會命中註解、字串常數、文件引用；`ls` 的路徑打錯會得到「不存在」的假象。派發者在寫 context 時處於「整理已知」的心態而非「驗證未知」，故不對自己的搜尋結果施加與執行者同等的懷疑。

**放大因素**：context 寫進 ticket 後具有權威外觀——它位於 Problem Analysis 這種語意上代表「已分析」的章節，且通常附帶行號、指令、數字。執行者的預設是採信而非複驗，除非實作中撞到矛盾。

**generic 化**：任何「A 為 B 準備輸入，B 依該輸入行動」的協作皆可重現——需求文件、API 契約、資料字典、交接文件。共同結構是「準備者的查證強度低於使用者的依賴強度」。

## 案例

flutter_balance 專案 v0.0.1，同一 session 內四次由執行者更正派發者的前提：

| Ticket | 派發者寫入 | 執行者查證結果 |
|--------|-----------|---------------|
| `0.0.1-W1-006` | `ticket-body-schema.md` 缺 1 項（`Spawn Requests`） | 實缺 4 項（另缺 `Context Bundle` / `NeedsContext` / `Exit Status`） |
| `0.0.1-W1-012` | 「該目錄下無既有 README」 | README 自初始 commit `f375ae6` 即存在，3779 bytes，且已載 `--with pyyaml` 用法 |
| `0.0.1-W1-012` | 「評估能否以依賴宣告取代手動旗標」 | `pyproject.toml` 已宣告 `pyyaml>=6.0` 兩處並有 `uv.lock`；真正根因是執行位置（repo 根無 `pyproject.toml`，`uv run` 建臨時環境忽略既有宣告） |
| `0.0.1-W1-015` | 「4 個檔含 PEP 723」 | 實為 3 個。第 4 個來自 `test_wrap_decision_tripwire_hook.py:204` 的 `assert "# /// script" in text`，為斷言字串非 metadata 區塊 |
| `0.0.1-W1-015` | 判準為「dev group 子集」 | 應為 `[project].dependencies` 與 dev group 的聯集——`uv run --directory` 安裝專案本身並預設啟用 dev group；實測 `pytest` 僅在 dev group 而環境中可用 |

第 4 例的後果具體：若偵測器採子字串比對，`test_wrap_decision_tripwire_hook.py` 會被當成 PEP 723 檔案送去解析，成為恆定假陽性。執行者改用 PEP 723 規範的行錨定正規式並將該情境固化為回歸測試。

四例中執行者皆主動查證並更正，未有前提錯誤流入產出。此為執行者品質的結果，非流程保證——若執行者採信前提，錯誤會直接固化。

## 防護

防護目標是使派發者的查證強度匹配執行者的依賴強度。

**寫入前的三類標註**：Context Bundle 的每項陳述須可歸入下列之一，無法歸類者不寫入。

| 標註 | 含義 | 要求 |
|------|------|------|
| 已驗證 | 附取得方式（指令與輸出） | 搜尋類須說明 pattern 的精確度考量 |
| 推測 | 明示為未驗證假設 | 附「執行者應獨立驗證，不採信本節推論」 |
| 未知 | 明示為待執行者判斷 | 不以「應該是 X」的形態陳述 |

**搜尋結果的精確度自問**：以 `grep` / `find` / `ls` 的輸出作為事實前，問「這個 pattern 會不會命中我不想要的東西」。

| 風險 | 檢查 |
|------|------|
| 未錨定子字串比對 | 命中註解、字串常數、文件引用？加 `^` 錨定或改結構化解析 |
| 斷言不存在 | 路徑是否打錯？換一種查法交叉確認（`ls` 與 `find` 與 `git log --` 併用） |
| 計數 | 抽樣檢視命中內容，勿只看 `wc -l` |

**否定命題加倍查證**：「X 不存在」「Y 未宣告」這類斷言的錯誤成本高於肯定命題——它會導向「建立 X」的工作方向，而正確方向是「更新既有 X」。斷言不存在前，至少以兩種方式確認。

**執行者側的對應責任**：ticket 的 Problem Analysis 已載明「接手者應獨立重新驗證數量／範圍（PC-162）」。本 pattern 是該要求在派發者側的鏡像——執行者複驗是安全網，派發者查證是主責，不應以前者存在為由降低後者強度。

## 相關

- 派發者側的 context 要求：`PC-040`（context 入 ticket 而非 prompt）——本 pattern 補其品質面，PC-040 管位置，本 pattern 管正確性
- 執行者側的複驗要求：`PC-162`（接手者應獨立重新驗證數量／範圍）
- 同源「宣稱與實際不符」家族：`PC-BAL-001`（驗證端清單過期）、`PC-BAL-002`（合併遺漏但檔頭宣稱完整）——三者皆為「文字陳述與實際狀態脫節」，本 pattern 的脫節發生在派發環節
- 工具輸出信任判據：`.claude/rules/core/tool-output-trust-rules.md` 規則 3（關鍵事實用固定值交叉驗證）——本 pattern 的搜尋精確度自問是其在 grep 類輸出上的具體化
