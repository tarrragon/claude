# PC-APP-011: 驗收者沿用執行者的偵測 pattern——假陰性驗收

## 症狀

清理/改標/遷移類任務的驗收宣告「已清零/無殘留」，但殘留物實際存在——因為驗收用的偵測 pattern 與執行者相同（或同樣狹窄），對 pattern 涵蓋範圍外的同類缺陷零覆蓋。

**識別特徵**：
- 驗收指令是從執行 agent 的 prompt 或自報結果複製而來的同一條 grep。
- 偵測 pattern 描述的是「已知變體」（如特定編號格式），而缺陷母體的變體空間未經窮舉論證。
- 多輪「清零宣告 → 新工具/新視角一掃又現一族」的循環。

## 實證案例（0.38.1 UC 治理批次，2026-07-14，一日三連發）

| 輪次 | 驗收 pattern | 假陰性內容 |
|------|-------------|-----------|
| 1 | agent 自報 grep `UC-008\|UC-001\.[0-9]`（其被指派範圍） | 同檔 `UC-006.2.6`×2（跨 domain 出現，pattern 外） |
| 2 | PM 復驗 grep `UC-0[0-9]{2}`（三位數數字族） | 具名子樹族 `UC-01.Enrich`/`UC-08.PERF` 等 **200 處/43 檔**全盲，兩度宣告「全 lib 清零」 |
| 3 | 全 token 掃描工具（`uc verify`，白名單反向邏輯） | 首跑即揭露上述整族——證明前兩輪驗收皆假陰性 |

## 根因

1. **執行者 pattern 天然狹窄**：agent 只對「被指派的變體」建立偵測，其自報驗證本質是「我改的地方我改完了」，不是「該類缺陷不存在了」。
2. **驗收者複用同 pattern 即繼承同盲區**：驗收的獨立性被誤解為「換人執行同一條指令」；維度不獨立，換人等於沒換。
3. **正向枚舉 vs 反向白名單的結構差**：枚舉已知壞變體（`UC-008|UC-001.x`）永遠追不上生成式 AI 可產生的變體空間；只有反向邏輯（「不在合法集內的一律違規」）能封閉缺陷母體。

## 解決方案

1. 驗收改標/清理類交付時，主線程使用**維度獨立**的交叉檢查，擇一以上：
   - 總量守恆對比：改標數學是否成立（實證：bare UC-01 136→162 = +17 UC-008 + 9 保留，數字對上才是真閉環）。
   - 反向白名單全掃描：以「合法集」反推違規，而非枚舉已知壞 pattern。
   - 專用驗證工具（若存在）取代手寫 grep。
2. acceptance 撰寫「工具終態」而非「pattern 無命中」：`doc uc verify lib` exit 0 這類反向邏輯終態，使工作發現與驗收共用同一把完整的尺。

## 預防措施

- 派發清理類 ticket 時，PM 在 acceptance 即寫入反向邏輯終態條件；無現成工具時，先問「這類缺陷的變體空間是否可枚舉」，不可枚舉則優先建反向驗證工具再清理。
- 驗收 checklist 加一問：「我的驗證指令與執行者的偵測 pattern 是否同源？」同源即重新設計驗證維度。
- 與既有原則銜接：「worktree 交付物為權威但主線程獨立復驗」的「獨立」指**維度獨立**，不只是執行者獨立。
