# PC-BAL-002: 檔案合併遺漏子功能，但檔頭仍宣稱完整承接

## 症狀

兩個模組合併為一時，部分子功能未被移植，但合併後檔案的檔頭 docstring 仍宣稱「合併自 X」而未標註遺漏項。原檔通常同時被解除註冊或標記為 superseded，於是：

- 該子功能在系統中完全失效——原檔有實作但不執行，後繼者會執行但無實作
- 失效不產生錯誤，只是某個判斷分支永遠不觸發，症狀是「該被抑制的沒被抑制」「該被分類的沒被分類」這類沉默劣化
- 後續維護者讀到檔頭的「合併自 X」宣稱，會將原檔判定為 dead code 而刪除，永久移除該子功能的唯一實作

## 根因

合併作業以「主流程可運作」為完成判準，未逐一盤點原檔的公開函式與常數。主流程通常由最常觸發的路徑構成，邊緣判斷（抑制條件、分類邏輯、特例處理）不在該路徑上，測試若只覆蓋主流程也不會紅燈。

檔頭宣稱是人工撰寫的自然語言，不受任何檢查約束。「合併自 X」在合併當下是意圖陳述，完成後未被校正為事實陳述，於是意圖與實際的落差被固化為文件。

**generic 化**：跨領域可重現於任何「多來源合併為單一實作」的作業——模組合併、服務整併、設定檔統一、函式庫升級時的相容層改寫。共同結構是「合併後的產物自述其來源，但無機制驗證該自述」。

## 案例

flutter_balance 專案 0.0.1-W1-003 與 0.0.1-W1-005。

`cli-error-feedback-hook.py:12` 檔頭載明「合併自 `skill-cli-error-feedback-hook.py`（ticket/skill CLI 錯誤偵測）」。實際比對函式清單：

| 子功能 | 原檔 | 後繼者（合併宣稱完成後） |
|--------|------|------------------------|
| `is_skill_cli_command` | 有 | 有 |
| `is_excluded_error` | 有 | 有 |
| `detect_skill_error_type` | 有 | 有 |
| `check_skill_cli_error` | 有 | 有 |
| `is_envelope_output` + `ENVELOPE_VERSION_MARKER`（W17-008.5.5） | 有（line 97/220/405） | 無 |
| `classify_error` / `detect_system_gap`（W3-073） | 有 | 無 |

四個主流程函式完整移植，兩組邊緣判斷遺漏。行為實測：以含 `__error_envelope_v1__` 標記與不含標記的兩組 payload 餵入後繼者，`additionalContext` 皆為 532 字元，證實抑制判斷不存在。

原檔未在 settings.json 註冊，故該功能在框架中處於雙重失效狀態。0.0.1-W1-003 原規劃將原檔作為 superseded dead code 刪除，刪除前的函式比對才揭露遺漏，據此撤回刪除並建票移植（0.0.1-W1-005 完成移植後刪除，`classify_error` 組另建 0.0.1-W1-008 決策）。

遺漏兩處而非一處，是判定其為系統性而非單點疏漏的依據。

## 防護

防護目標是使「合併不完整」在合併當下就暴露，而非在數月後的刪除作業中偶然被發現。

**合併完成判準**：主流程可運作不等於合併完成。合併前列出原檔的全部公開函式與模組級常數，逐一標註「已移植 / 刻意棄用（附理由）」，兩者以外的狀態不得視為完成。

| 訊號 | 判定 | 動作 |
|------|------|------|
| 檔頭宣稱「合併自 X」但無遺漏項標註 | 宣稱未經驗證 | 比對原檔公開介面後才採信 |
| 原檔存在但未註冊 / 未被 import | 該檔功能全面失效 | 刪除前必須比對介面，不得僅憑檔頭宣稱判定 dead code |
| 合併後測試全綠 | 不構成完整性證據 | 測試覆蓋主流程，邊緣判斷需另行確認 |

**刪除前的強制比對**：判定某檔為 superseded 而準備刪除時，比對其公開介面與宣稱的後繼者。此檢查成本低（一次 `grep -c "def "` 對照）且是遺漏的最後攔截點——一旦刪除，唯一實作消失。

**檔頭校正**：合併完成後將檔頭的意圖陳述改為事實陳述，明列已承接與未承接項。未承接項附理由與追蹤 ticket ID，使後續維護者不需重新調查。
