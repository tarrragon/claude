# PC-V1-012: 防護置於便利攔截介面而非狀態變異的真實源頭

## 基本資訊

- **Pattern ID**: PC-V1-012
- **分類**: process-compliance（設計推理失誤）
- **風險等級**: 中
- **關聯**: [[ARCH-TUNL-001]]（local hook 幽靈：症狀與單一註冊來源解法）、[[ARCH-V1-001]]（單點執法多入口繞過）、PC-148（hook 雙重註冊）

## 錯誤症狀

為防止某個壞狀態（幽靈註冊、殘缺資料、非法寫入）復發而設計 guard 時，把攔截點放在「對開發者最直覺、最便利的介面」，而非「該壞狀態實際產生的源頭」。典型表現：

- 提案以 PreToolUse / 手動編輯攔截為治本層，但該壞狀態的真實復發路徑是**自動化流程**（同步、安裝、批次生成、遷移腳本），手動介面只是罕見路徑。
- premortem 自己寫出「既有資料未先遷移就上 guard → 自我阻擋」一類矛盾，卻仍把該方案列為治本。
- 把「攔截未來寫入」（行為防護）誤當「狀態不該存在」（結構治本）；guard 只擋新增、不回收存量，且 per-scope（單專案 / 單入口）無法覆蓋問題的真實作用域（框架層 / 所有路徑）。

## 根因分析

### 根因 1：攻擊面辨識錯誤——便利介面 ≠ 真實變異源頭

設計者從「我最常怎麼碰到這個檔」反推攔截點，而非從「這個壞狀態實際由哪條路徑寫入」反推。當真實寫入源是自動化流程（人不直接經手），裝在手動編輯介面的 guard 攔到的是幾乎不會發生的路徑，真實源頭照舊繞過。

### 根因 2：行為防護冒充結構治本

「攔截寫入」屬行為層（擋動作），「狀態不該存在 / 真值只有單一來源」屬結構層（消除壞狀態的存在條件）。兩者被混為一談時，方案在「治本」名義下只做了行為攔截：對存量無效、對其他作用域無效，根本壞狀態仍可由未被攔的源頭再生。

### 根因 3：premortem 訊號被忽略

行前預想已暴露「自我阻擋 / 存量未回收 / 作用域落差」等矛盾，但因方案已被命名為「治本層【採】」而未回頭推翻。命名錨定（Anchor）讓矛盾訊號被當作待緩解的細節，而非否決依據。

## 解決方案

### 步驟 1：先定位真實變異源頭

問「這個壞狀態實際由哪條路徑寫入？人手動寫的頻率 vs 自動化寫的頻率？」把 enforcement 放在**所有寫入路徑的共同瓶頸**或**自動化源頭**，而非便利攔截點。對照 [[ARCH-V1-001]]「驗證下沉至所有路徑共同瓶頸」。

### 步驟 2：區分行為防護與結構治本

| 層 | 問題 | 對存量 | 對其他作用域 |
|----|------|--------|-------------|
| 行為防護（攔截寫入） | 擋未來動作 | 無效（需另做遷移） | per-scope，不覆蓋 |
| 結構治本（消除存在條件） | 真值單一來源 / 壞狀態無法表達 | 一次到位 | 全作用域 |

宣稱「治本」前先確認方案落在結構層；若只能做行為攔截，誠實標為「安全網 / 行為防護」，不冒充治本。

### 步驟 3：把 premortem 矛盾當否決訊號

行前預想若出現「自我阻擋 / 存量未回收 / 作用域落差」，回 WRAP W 階段重擴增，尋找作用在源頭或結構層的方向，而非緩解便利攔截方案的副作用。

## 預防措施

- 多視角審查（parallel-evaluation 情境 F）對「防護放置層級」設專門檢查：攔截點是否在真實變異源頭？是行為防護還是結構治本？premortem 矛盾是否被當否決？
- 設計 guard 的 ticket 在 Solution 明列「真實寫入源頭路徑盤點」一節，否則審查退回。

## 通用化檢驗

將本案專案細節（settings.local.json / sync-pull / PreToolUse）替換為通用描述後，敘述仍成立：「為防止壞狀態復發而設計 guard，把攔截點放在便利介面而非該壞狀態的自動化真實源頭，導致攔錯攻擊面、對存量與其他作用域無效，並把行為防護誤當結構治本。」→ 屬跨專案可重現的結構性設計推理失誤。
