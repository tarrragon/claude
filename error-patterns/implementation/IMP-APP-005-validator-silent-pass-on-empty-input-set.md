# IMP-APP-005: 驗證工具對空輸入集假綠燈

## 症狀

驗證類工具（linter / verify 命令 / CI gate / hook）在「根本沒有掃到任何東西」時回報「驗證通過」exit 0——路徑打錯、白名單解析失敗、glob 無匹配等輸入層故障被當成「零違規」。

**識別特徵**：
- 掃描函式對無效輸入（不存在路徑、空定義集）`return []`，下游把空違規清單直譯為 pass。
- 「驗證通過」訊息不附「掃描了多少檔案 / 白名單有多少項」等分母資訊。
- CI 中該步驟從未紅過，且無人能說出它上次真正攔到東西是何時。

## 實證案例（0.38.1-W1-064 uc verify，2026-07-14 平行審查發現）

`doc uc verify /nonexistent/path` → 輸出「UC 編號驗證通過，無違規 token。」exit 0。根因：`_iter_scannable_files` 對 `not root.is_dir()` 靜默回空清單，違規判定端把「無檔案」與「有檔案且全合規」折疊為同一狀態。CI 掛載時打錯路徑即永久假綠。同族：白名單（spec 解析）為空時，語意應為 fail-fast 而非「全部違規」或「全部通過」。

## 根因

1. 「無違規」與「無輸入」是兩個狀態，被單一空清單資料型別折疊——型別上 `[]`（掃了但乾淨）與 `[]`（沒掃到東西）不可區分。
2. 驗證工具的 happy path 思維：測試只覆蓋「有輸入有違規」與「有輸入無違規」，輸入層故障（第三態）無測試。
3. exit code 語意設計時只分 pass/violation 兩態，未保留「工具自身故障」的第三值。

## 解決方案

1. exit code 三態：`0`=掃描有效且無違規 / `1`=有違規 / `2`=輸入層故障（壞路徑、空白名單、解析失敗），故障訊息附診斷（路徑、白名單大小）。
2. pass 訊息附分母：「掃描 N 檔、白名單 M 項、0 違規」——分母為 0 即自我暴露。
3. 輸入驗證前置 fail-fast：路徑不存在、定義集為空，在掃描前即報錯。

## 預防措施

- 撰寫/審查驗證類工具時固定三問：「輸入不存在時它說什麼？」「定義集為空時它說什麼？」「pass 輸出能否看出它真的掃了東西？」
- 測試設計必含輸入層故障案例（壞路徑/空定義集/無匹配 glob）。
- 與 PC-APP-011 銜接：假綠燈是「驗收假陰性」的工具內建版——pattern 盲區在驗收層，空輸入折疊在工具層，兩者都讓「通過」失去證據力。
