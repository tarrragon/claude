# Opinionated Default 設計原則（完整版）

> **速查 stub**：`.claude/rules/core/opinionated-default-design.md`（自動載入）。本檔為按需讀取的完整論證。

工具的預設行為就是團隊的實際流程。文件規範和工具預設不一致時，工具會贏——人類走阻力最小路徑，AI agent 更是只走預設路徑。設計工具、規則、方法論、流程時，預設路徑必須引導正確做法。

> **來源**：版本錯置事件復盤。改善類 ticket 被放入新功能版本（CLI 接受任何版本號不提示）、已完成版本殘留 active 狀態（發布工具不掃描前版本）。兩者根因相同：工具在應該有立場的地方選擇了沉默。

---

## 核心主張

### 主張 1：預設行為 > 文件規範

**工具的預設路徑就是使用者的實際行為。** 文件定義「應該是什麼樣」，工具實現「實際是什麼樣」。兩者不一致時，優先修工具。

**Why**：使用者（人類和 AI）都走阻力最小路徑。文件需要記住、查閱、主動遵守；工具預設行為零成本自動生效。

**Consequence**：每一個「寫文件提醒使用者遵守操作規範」都是信號——工具的預設行為還有改善空間。看到這個信號時，優先評估能否把提醒轉化為工具的預設行為。

### 主張 2：存在「多數情況下正確」的預設值時，工具必須表達

**判斷標準**：對每個使用者需要做選擇的地方，問「有沒有多數情況下正確的預設值？」有的話，工具應表達建議式 opinion（提示預設 + 允許覆蓋）。

**Why**：建議錯了，使用者覆蓋一次，幾秒鐘；沉默錯了，事後修正，幾小時。只要建議的正確率不是極低，建議就比沉默划算。錯誤成本不對稱決定了形式。

### 主張 3：AI agent 使用者時 opinion 從建議升級為必要

**Why**：人類有跨次記憶（「上次放錯被糾正過，這次注意」）；AI agent 每個 session 全新，沒有跨 session 記憶。除非教訓被固化成工具行為，否則每次都重犯。工具在操作當下的即時引導是離決策點最近的攔截——在 AI agent 場景，這是唯一可靠防線。

### 主張 4：清理路徑必須主動掃描

**工具不只做「被要求做的事」，還要做「應該做的事」。** 前一次操作應該同步但沒同步的狀態，工具必須主動掃描和報告。

**Why**：殘留狀態不影響當前操作但誤導後續所有查看者。每次操作都是修正殘留的機會窗口——錯過就一直留著。

## Opinion 的兩種形式

| 形式 | 適用場景 | 範例 |
|------|---------|------|
| 建議式（提示 + 允許覆蓋） | 歸屬判斷、分類選擇、預設值 | CLI 提示建議版本，`--version` 覆蓋 |
| 阻擋式（需額外操作才能繞過） | 高風險操作、不可逆動作 | force push 需 `--force` flag |

**判準**：錯誤成本不對稱決定形式。歸屬判錯可事後修正 → 建議式；資料損毀不可逆 → 阻擋式。

## 適用範圍

本原則不限於 CLI 工具，適用於所有「使用者需要做選擇」的設計：

| 設計對象 | 應用方式 |
|---------|---------|
| CLI 工具 | 參數預設值、建議提示、pre-flight 掃描 |
| Skill 設計 | 工作流預設路徑、錯誤處理預設行為 |
| Hook 設計 | 預設觸發行為、預設通知層級 |
| 規則 / 方法論 | 若需寫「請遵守 X」→ 先評估能否改為工具預設行為 |
| Agent 定義 | 預設職責邊界、預設產出格式 |
| API / 表單 | 預設參數值、預選值、必填驗證 |
| 自動化流程 | 預設路由、預設分支 |

## 反模式 → 正確做法對照

| 反模式（寫文件） | 正確做法（改工具） |
|----------------|-------------------|
| README 寫「請先跑測試」 | pre-commit hook 自動跑 |
| 貢獻指南寫 PR 範本 | PR template 預填結構 |
| 版本策略文件寫規則 | CLI 根據 ticket type 建議版本 |
| 文件寫「production 需額外確認」 | API 預設 staging，production 需顯式指定 |
| 說明文字寫「建議填寫」 | 欄位預設值 + 必填驗證 |
| 規則寫「派發前讀 context bundle」 | Hook 自動檢查 context bundle 是否存在 |
| 方法論寫「Phase 4 不可跳過」 | acceptance-gate-hook 自動阻擋缺 Phase 4 的 complete |

## 設計檢查清單

設計或審查任何工具 / Skill / CLI / Hook / 流程時：

1. **列出所有使用者需要做選擇的地方**（參數、欄位、路徑、分支）
2. **對每個問**：有沒有「多數情況下正確」的預設值或建議值？
3. **有的話**：加建議式 opinion（提示預設 + 允許覆蓋）
4. **檢查清理路徑**：有沒有前一次操作應該同步但沒有同步的狀態？
5. **AI agent 加倍**：如果工具會被 AI agent 或自動化流程呼叫，上述每一項的優先級加倍
6. **信號掃描**：現有文件中有沒有「請遵守 X」「建議做 Y」的提醒？每一個都是工具預設行為的改善機會

## 案例

### 案例 1：改善類 ticket 放錯版本

- **情境**：`ticket create --version 0.4.0 --type ANA` — CLI 知道這是分析類 ticket，也知道 v0.4.0 是新功能版本。但 CLI 不判斷「分析類放新功能版本是否合理」
- **後果**：三張改善類 ticket 和 PostgreSQL 功能混在一起，事後遷移花一輪操作成本
- **修正**：CLI 根據 ticket type 建議 patch 版本，使用者可用 `--version` 覆蓋

### 案例 2：已完成版本殘留 active

- **情境**：版本發布工具只檢查「當前版本 ticket 是否完成」，不掃描「前版本是否有 active 殘留」
- **後果**：三個已完成版本在看板顯示為「有未完成工作」，持續誤導
- **修正**：發布工具 pre-flight 加前版本狀態掃描，發現殘留即 WARN

### 案例 3：框架中的 opinion 實踐

本專案已有的 opinionated default 實踐：

| 工具 | opinion 表達 |
|------|-------------|
| acceptance-gate-hook | Phase 4 缺失 → 阻擋 complete（阻擋式） |
| branch-status-reminder | 在保護分支 → 建議建 worktree（建議式） |
| dispatch-hook | context bundle 缺失 → 警告（建議式） |
| pre-commit hooks | 測試未通過 → 阻擋 commit（阻擋式） |

## 與其他規則的邊界

| 規則 | 邊界 |
|------|------|
| `quality-baseline.md` 規則 4（Hook 失敗必須可見） | 本原則處理「預設行為設計」；規則 4 處理「失敗時的可見性」。互補不重疊 |
| `tool-discovery.md`（宣告做不到前五問） | 本原則處理「工具該有什麼行為」；tool-discovery 處理「工具能力是否存在」 |
| `ai-communication-rules.md` 規則 6（估時驅動） | 本原則的「建議式 opinion」是預設值設計，非估時判斷 |
| `skill-design-guide` §1.3（自由度匹配脆弱性） | §1.3 決定「自由度高低」；本原則決定「預設路徑是否表達 opinion」。高自由度的場景仍可有預設建議 |
